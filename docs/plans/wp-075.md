# WP-075 Plan: SABSA Bidirectional Traceability

**Date:** 2026-04-03
**Status:** Ready for implementation

---

## Summary

Add four read-only traceability endpoints to the knowledge layer:
`GET /knowledge/controls/{id}/trace-up`, `GET /knowledge/controls/{id}/trace-down`,
`GET /knowledge/attributes/{id}/coverage`, and `POST /knowledge/gap-analysis`.
Together these expose the SABSA bidirectional trace — from a control up to business
attributes and norms, and down to evidence documents and Memory nodes — plus a
three-way reconciliation gap analysis across multiple controls.

---

## Spec Reconciliation Against ADR-002

The WP spec was written before ADR-002 finalised the graph model. Two paths in the spec
need correcting before any Cypher is written.

**Trace-up path in spec:** "Control → Framework → BusinessAttribute via SABSA_LAYER edges → Norm via IMPLEMENTS edges"

**Actual ADR-002 model:**
- There is no `SABSA_LAYER` edge. BusinessAttribute is reached via Precepts:
  `Control -[ADDRESSES]-> Precept -[FULFILS]-> BusinessAttribute`
- Norms reach Precepts via `Norm -[REQUIRES]-> Precept` (not IMPLEMENTS to BusinessAttribute)
- `MAPS_TO` goes Norm → Control (not the other direction)
- `IMPLEMENTS` goes Document → Control (evidence claim, not norm-to-attribute)

**Resolution:** Trace-up Cypher will follow the correct ADR-002 path:
```
Control -[ADDRESSES]-> Precept -[FULFILS]-> BusinessAttribute
Norm -[REQUIRES]-> same Precept
```

The control hierarchy traversal `(CONTAINS)*` is correct as specified — walk up the
CONTAINS tree to find top-level Controls that ADDRESSES a Precept (lower-level Controls
inherit through tree traversal per ADR-002).

## Function Signatures (repo)

```python
def trace_up(session, control_id: str) -> dict | None:
    """Return {control_id, business_attributes, norms} or None if control not found."""

def trace_down(session, control_id: str, org_id: str | None) -> dict | None:
    """Return {control_id, documents, evidence_memories, gap_memories} or None if not found.

    org_id filters ABOUT_CONTROL edges; when None, returns all edges (universal).

    """Return {attribute_id, total_controls, covered_controls, coverage_pct,

    session,
    control_ids: list[str],
    org_id: str | None,
) -> dict:
    """Three-way reconciliation: {covered, partial, uncovered}.
    If control_ids is empty, analyses all controls.
    """
```

## Cypher Query Designs

### 1. `trace_up`

Walk the CONTAINS tree upward to find Controls that have an ADDRESSES edge,
then follow Precept → BusinessAttribute and collect Norms that REQUIRE the same Precept.

```cypher
// Step 1: find the control
MATCH (c:Control {id: $control_id})

// Step 2: walk up the CONTAINS tree to ancestors (including self)
// Memgraph supports variable-length paths; direction is parent -[CONTAINS]-> child
// so we walk backwards: child <-[CONTAINS]- parent
WITH c
MATCH path = (root:Control)-[:CONTAINS*0..]->(c)
WHERE NOT ()-[:CONTAINS]->(root)   // root has no incoming CONTAINS
WITH collect(DISTINCT root) + [c] AS ancestors

// Step 3: for each ancestor that ADDRESSES a Precept, collect attributes and norms
UNWIND ancestors AS anc
OPTIONAL MATCH (anc)-[:ADDRESSES]->(p:Precept)-[:FULFILS]->(ba:BusinessAttribute)
OPTIONAL MATCH (n:Norm)-[:REQUIRES]->(p)
RETURN
  $control_id AS control_id,
  collect(DISTINCT {id: ba.id, name: ba.name}) AS business_attributes,
  collect(DISTINCT {id: n.id, name: n.name, status: n.status}) AS norms
```

**Note on Memgraph gotcha:** The variable-length path `[:CONTAINS*0..]` traversal must be
tested against a live instance to confirm performance on deep trees. The `WHERE NOT ()-[:CONTAINS]->(root)` 
root-detection clause depends on no orphan CONTAINS edges existing above root Controls.
If this clause causes issues in practice, replace with a max-depth bound (e.g. `*0..10`).

**Simplified single-query approach** (preferred — avoids collect-then-unwind):

```cypher
MATCH (c:Control {id: $control_id})
OPTIONAL MATCH (anc:Control)-[:CONTAINS*0..]->(c)
  WHERE NOT ()-[:CONTAINS]->(anc)
WITH c, coalesce(anc, c) AS root
OPTIONAL MATCH (root)-[:ADDRESSES]->(p:Precept)-[:FULFILS]->(ba:BusinessAttribute)
OPTIONAL MATCH (n:Norm)-[:REQUIRES]->(p)
RETURN
  $control_id AS control_id,
  collect(DISTINCT {id: ba.id, name: ba.name}) AS business_attributes,
  collect(DISTINCT {id: n.id, name: n.name, status: n.status}) AS norms
```

**Risk:** The root-ancestor pattern may be slow on deep trees. If Memgraph has issues with
the negation in WHERE, fall back to: return all ancestors including self and collect
Precepts from all levels (controls at any level may ADDRESSES a Precept directly).

**Pragmatic fallback** (include all levels, not just root):

```cypher
MATCH (c:Control {id: $control_id})
OPTIONAL MATCH (anc:Control)-[:CONTAINS*0..]->(c)
WITH c, collect(DISTINCT anc) + [c] AS all_ancestors
UNWIND all_ancestors AS a
OPTIONAL MATCH (a)-[:ADDRESSES]->(p:Precept)-[:FULFILS]->(ba:BusinessAttribute)
OPTIONAL MATCH (n:Norm)-[:REQUIRES]->(p)
RETURN
  $control_id AS control_id,
  collect(DISTINCT CASE WHEN ba IS NOT NULL THEN {id: ba.id, name: ba.name} END) AS business_attributes,
  collect(DISTINCT CASE WHEN n IS NOT NULL THEN {id: n.id, name: n.name, status: n.status} END) AS norms
```

The repo function should filter out `None` entries from the collected lists (Cypher
`collect(DISTINCT ...)` with CASE can include null-equivalent maps).

**Implementation decision:** use the pragmatic fallback (collect from all ancestors, not
just root) since it is more complete and avoids the root-detection edge case. Post-process
in Python to de-duplicate and strip None entries.

### 2. `trace_down`

```cypher
MATCH (c:Control {id: $control_id})

// Documents: Chunk -[SUPPORTS]-> Control, then Document -[HAS_CHUNK]-> Chunk
OPTIONAL MATCH (ch:Chunk)-[sup:SUPPORTS]->(c)
OPTIONAL MATCH (d:Document)-[:HAS_CHUNK]->(ch)

// Memory evidence/gaps (always OPTIONAL — must work with zero Memory nodes)
OPTIONAL MATCH (m:Memory)-[r:ABOUT_CONTROL]->(c)
  WHERE r.relationship_type IN ["evidence", "gap"]
    AND ($org_id IS NULL OR r.org_id = $org_id OR r.org_id IS NULL)

RETURN
  $control_id AS control_id,

  // group chunks under their documents
  collect(DISTINCT CASE WHEN d IS NOT NULL THEN {
    doc_id: d.id,
    doc_title: d.title,
    chunk_id: ch.id,
    chunk_text: ch.text,
    confidence: sup.confidence,
    status: sup.status
  } END) AS doc_chunks,

  collect(DISTINCT CASE WHEN m IS NOT NULL THEN {
    id: m.id,
    text: m.text,
    relationship_type: r.relationship_type
  } END) AS memory_refs
```

**Python post-processing in repo function:**
- Group `doc_chunks` by `doc_id` to build `DocumentWithChunks` list
- Split `memory_refs` into `evidence_memories` (relationship_type="evidence") and
  `gap_memories` (relationship_type="gap")
- Filter None entries from all collections

**Org-id scoping logic:** When `org_id` is provided, the filter
`r.org_id = $org_id OR r.org_id IS NULL` includes universal edges (null org_id = applies
to all orgs). When `org_id` is None (not provided), all ABOUT_CONTROL edges are returned
regardless of their org_id.

### 3. `attribute_coverage`

```cypher
// Find the BusinessAttribute and all Controls that Precepts bridge to it
MATCH (ba:BusinessAttribute {id: $attribute_id})
MATCH (ba)<-[:FULFILS]-(p:Precept)<-[:ADDRESSES]-(ctrl:Control)
WITH ba, collect(DISTINCT ctrl) AS all_controls

// For each control, check if at least one SUPPORTS chunk exists
UNWIND all_controls AS ctrl
OPTIONAL MATCH (ch:Chunk)-[:SUPPORTS]->(ctrl)
WITH ba, ctrl, count(ch) AS chunk_count

RETURN
  ba.id AS attribute_id,
  count(ctrl) AS total_controls,
  sum(CASE WHEN chunk_count > 0 THEN 1 ELSE 0 END) AS covered_controls,
  collect(CASE WHEN chunk_count = 0 THEN ctrl.id END) AS uncovered_control_ids
```

**Python post-processing:** Calculate `coverage_pct = covered_controls / total_controls * 100`
(guard against divide-by-zero when total_controls = 0). Filter None from
`uncovered_control_ids`.

**Return None from repo function** if the MATCH on BusinessAttribute returns no record
(attribute not found). Routes layer raises HTTP 404.

### 4. `gap_analysis`

This is the most complex endpoint. Uses two separate queries:
1. Resolve `control_ids` (if empty, fetch all Control ids)
2. Run the three-way classification query

**Query A: resolve all control ids if input is empty**

```cypher
MATCH (c:Control)
RETURN c.id AS id, c.name AS name
```

**Query B: classify each control**

```cypher
UNWIND $control_ids AS cid
MATCH (c:Control {id: cid})

// Check for SUPPORTS chunks
OPTIONAL MATCH (ch:Chunk)-[:SUPPORTS]->(c)

// Check for evidence memories (always OPTIONAL)
OPTIONAL MATCH (m:Memory)-[r:ABOUT_CONTROL]->(c)
  WHERE r.relationship_type = "evidence"
    AND ($org_id IS NULL OR r.org_id = $org_id OR r.org_id IS NULL)

RETURN
  c.id AS control_id,
  c.name AS control_name,
  count(DISTINCT ch) > 0 AS has_chunks,
  count(DISTINCT m) > 0 AS has_evidence_memories
```

**Python classification in repo function:**

```python
covered, partial, uncovered = [], [], []
for row in results:
    entry = {
        "control_id": row["control_id"],
        "control_name": row["control_name"],
        "has_chunks": row["has_chunks"],
        "has_evidence_memories": row["has_evidence_memories"],
    }
    if row["has_chunks"] and row["has_evidence_memories"]:
        covered.append(entry)
    elif not row["has_chunks"] and not row["has_evidence_memories"]:
        uncovered.append(entry)
    else:
        partial.append(entry)
return {"covered": covered, "partial": partial, "uncovered": uncovered}
```

**Cypher note:** Memgraph's `count(DISTINCT ch) > 0` in a RETURN clause should work.
If boolean expressions in RETURN cause issues, return `count(DISTINCT ch)` and
`count(DISTINCT m)` as integers and convert to bool in Python post-processing.

### Acceptance Criteria

Verifiable against the live service (manual smoke test or script):

1. `GET /knowledge/controls/{id}/trace-up` for a control with Precepts and BusinessAttributes
   → returns non-empty `business_attributes` and `norms` lists

2. `GET /knowledge/controls/{id}/trace-up` for a control with no Precept links
   → returns `{"control_id": ..., "business_attributes": [], "norms": []}` (not 500)

3. `GET /knowledge/controls/{id}/trace-down` with no Memory nodes in the graph
   → returns `{"documents": [...], "evidence_memories": [], "gap_memories": []}` (not 500)
   This is the **knowledge-only mode** proof required by ADR-001.

4. `GET /knowledge/controls/{id}/trace-down?org_id=org-acme` with one evidence Memory linked
   via `ABOUT_CONTROL {org_id: "org-acme"}` and one linked via `ABOUT_CONTROL {org_id: null}`
   → both appear in `evidence_memories` (universal edges always included)

5. `GET /knowledge/controls/{id}/trace-down?org_id=org-acme` with a Memory linked via
   `ABOUT_CONTROL {org_id: "org-other"}`
   → that Memory does NOT appear in `evidence_memories`

6. `GET /knowledge/attributes/{id}/coverage` → returns correct `coverage_pct`,
   `total_controls`, and `covered_controls`

7. `POST /knowledge/gap-analysis {}` (empty body, all controls) → returns all three lists;
   no 500 error even if no Memory nodes exist

8. `POST /knowledge/gap-analysis {"control_ids": ["c-1", "c-2"]}` → analyses only
   the specified controls

9. `GET /knowledge/controls/nonexistent/trace-up` → 404
10. `GET /knowledge/attributes/nonexistent/coverage` → 404

## Risks / Open Questions

### Risk 1: Ancestor traversal performance
The `(anc:Control)-[:CONTAINS*0..]->(c)` variable-length path traversal could be slow
on deep control trees. Mitigate in the short term by collecting from all ancestors
(not just root) — this avoids the `NOT ()-[:CONTAINS]->(root)` anti-pattern which may
force a full scan. If performance is a concern, add a depth bound (`*0..10`).

### Risk 2: Cypher boolean in RETURN
`count(DISTINCT ch) > 0 AS has_chunks` in a RETURN clause may behave differently across
Memgraph versions. If it fails, return `count(DISTINCT ch) AS chunk_count` and
`count(DISTINCT m) AS memory_count`, then compute booleans in Python.

### Risk 3: DETACH DELETE gotcha does not apply
These are all read-only queries. The gotcha about DETACH DELETE not supporting RETURN
is not applicable.

### Risk 4: gap-analysis with empty control_ids list — two queries vs one
Running `MATCH (c:Control) RETURN c.id, c.name` to expand an empty list, then running
the classification query with all IDs, is correct but creates a round-trip. This is
acceptable at MVP scale. A single combined query is an optimisation for WP-076 or later.

### Risk 5: ADR-002 scope of ADDRESSES edge
ADR-002 states "Top-level Controls address Precepts. Lower-level Controls inherit through
tree traversal." This means the `ADDRESSES` edge is only written on top-level Controls.
The trace-up query must collect Precepts from all ancestors, not just the immediate control.
The pragmatic fallback Cypher handles this correctly by collecting from `all_ancestors`.

### Open Question: `trace_down` control not found
The spec says trace-down returns documents, chunks, and memories. If the control doesn't
exist, the MATCH will produce no rows. The repo function should detect this (count of
rows = 0 when MATCH returns nothing) and return `None` to trigger a 404. Implementation:
run a separate `MATCH (c:Control {id: $control_id}) RETURN c.id` check first, then run
the main query — or check if the RETURN row count is zero and the control MATCH produced
nothing (since OPTIONAL MATCH on docs/memories will still return one row with nulls for
an existing control with no evidence).

**Recommended approach:** Run an existence check (`get_control`) before the main trace-down
query to keep the code simple and match the existing repo pattern.
