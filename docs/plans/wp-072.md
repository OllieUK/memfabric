# WP-072 Plan: Knowledge Bridge Module — Cross-Layer Memory Edges

**Date:** 2026-04-03
**Status:** Ready for implementation

---

## Summary

Create `knowledge_bridge.py` as the single module that couples episodic memory to the knowledge layer, adding `ABOUT_CONTROL` and `CITES_DOC` cross-layer edges. Extend `AddMemoryRequest`, `UpdateMemoryRequest`, and `MemoryHit` with knowledge-layer fields, guard all new behaviour behind `ENABLE_KNOWLEDGE_LAYER`, and wire the bridge into the `add_memory`, `update_memory`, and `merge_memory` routes. Extend MCP tools `memory_add` and `memory_update` with the new parameters.

## Function signatures

### `knowledge_bridge.py` — new functions

```python
def link_controls(
    session,
    memory_id: str,
    control_ids: list[str],
    relationship_type: str | None,
    org_id: str | None,
) -> None:
    """MERGE ABOUT_CONTROL edges from Memory to each Control.
    Silently skips control_ids that do not exist (no HTTP error — existence is
    pre-validated by validate_controls before this is called).
    """

def validate_controls(
    session,
    control_ids: list[str],
) -> list[str]:
    """Return control_ids that do NOT exist in the graph.
    Callers raise HTTP 400 if the returned list is non-empty.
    """

def link_documents(
    session,
    memory_id: str,
    doc_ids: list[str],
) -> None:
    """MERGE CITES_DOC edges from Memory to each Document.
    Silently skips doc_ids that do not exist (existence is pre-validated by
    validate_documents before this is called).
    """

def validate_documents(
    session,
    doc_ids: list[str],
) -> list[str]:
    """Return doc_ids that do NOT exist in the graph.
    Callers raise HTTP 400 if the returned list is non-empty.
    """

def replace_control_edges(
    session,
    memory_id: str,
    control_ids: list[str],
    relationship_type: str | None,
    org_id: str | None,
) -> None:
    """Delete all ABOUT_CONTROL edges from this Memory, then recreate for control_ids.
    Called by update_memory route when control_ids is provided.
    """

def replace_doc_edges(
    session,
    memory_id: str,
    doc_ids: list[str],
) -> None:
    """Delete all CITES_DOC edges from this Memory, then recreate for doc_ids.
    Called by update_memory route when doc_ids is provided.
    """

def rewire_cross_layer_edges(
    session,
    source_id: str,
    target_id: str,
) -> None:
    """Rewire ABOUT_CONTROL and CITES_DOC edges from source to target Memory.
    Called by merge_memory route after memory_repo.merge_memory completes.
    ABOUT_CONTROL: MERGE on (tgt, control) topology, keep relationship_type and
    org_id from source edge (ON CREATE only — if edge already exists on target,
    preserve target's properties).
    CITES_DOC: MERGE on (tgt, doc) topology.
    Both: DELETE source edge after MERGE.
    """

def hydrate_controls_and_documents(
    session,
    memory_ids: list[str],
) -> dict[str, dict]:
    """For each memory_id, return its linked controls and documents.
    Returns: {memory_id: {"controls": [...], "documents": [...]}}
    Used to populate MemoryHit.controls and MemoryHit.documents after search.
    """
```

## Cypher query designs

### 1. MERGE `ABOUT_CONTROL` edge (in `link_controls`)

```cypher
MATCH (m:Memory {id: $memory_id})
MATCH (c:Control {id: $control_id})
MERGE (m)-[r:ABOUT_CONTROL]->(c)
ON CREATE SET
    r.relationship_type = $relationship_type,
    r.org_id            = $org_id
```

Parameters: `$memory_id`, `$control_id`, `$relationship_type` (nullable), `$org_id` (nullable).

Called once per control_id in a Python loop — same pattern as strand/person linking in `add_memory`.

ON CREATE only: if the edge already exists (e.g. re-add after dedup reinforce), preserve existing properties. No ON MATCH clause needed.

### 2. Validate Control existence (in `validate_controls`)

```cypher
UNWIND $control_ids AS cid
OPTIONAL MATCH (c:Control {id: cid})
WITH cid, c
WHERE c IS NULL
RETURN cid AS missing_id
```

Parameters: `$control_ids` (list).

Returns rows for each missing ID. If any rows returned, caller raises HTTP 400.

### 3. MERGE `CITES_DOC` edge (in `link_documents`)

```cypher
MATCH (m:Memory {id: $memory_id})
MATCH (d:Document {id: $doc_id})
MERGE (m)-[:CITES_DOC]->(d)
```

Parameters: `$memory_id`, `$doc_id`.

No properties on `CITES_DOC` per ADR-002. Called once per doc_id.

### 4. Validate Document existence (in `validate_documents`)

```cypher
UNWIND $doc_ids AS did
OPTIONAL MATCH (d:Document {id: did})
WITH did, d
WHERE d IS NULL
RETURN did AS missing_id
```

Same pattern as control validation.

### 5. Delete + recreate `ABOUT_CONTROL` in update (in `replace_control_edges`)

```cypher
-- Step A: delete existing edges
MATCH (m:Memory {id: $memory_id})-[r:ABOUT_CONTROL]->(:Control)
DELETE r

-- Step B: recreate (per control_id in Python loop, same as link_controls)
MATCH (m:Memory {id: $memory_id})
MATCH (c:Control {id: $control_id})
MERGE (m)-[r:ABOUT_CONTROL]->(c)
ON CREATE SET
    r.relationship_type = $relationship_type,
    r.org_id            = $org_id
```

This is the same delete-and-recreate pattern used for `person_ids` and `strand_ids` in `update_memory` (lines 641–673 of `memory_repo.py`).

### 6. Delete + recreate `CITES_DOC` in update (in `replace_doc_edges`)

```cypher
-- Step A: delete existing edges
MATCH (m:Memory {id: $memory_id})-[r:CITES_DOC]->(:Document)
DELETE r

-- Step B: recreate (per doc_id in Python loop)
MATCH (m:Memory {id: $memory_id})
MATCH (d:Document {id: $doc_id})
MERGE (m)-[:CITES_DOC]->(d)
```

### 7. Rewire `ABOUT_CONTROL` in merge (in `rewire_cross_layer_edges`)

```cypher
MATCH (src:Memory {id: $src_id})-[r:ABOUT_CONTROL]->(c:Control)
MATCH (tgt:Memory {id: $tgt_id})
MERGE (tgt)-[existing:ABOUT_CONTROL]->(c)
ON CREATE SET
    existing.relationship_type = r.relationship_type,
    existing.org_id            = r.org_id
DELETE r
```

Parameters: `$src_id`, `$tgt_id`.

ON CREATE only (same rationale as ABOUT rewiring in `merge_memory` — if target already has the edge, keep target's properties). No property reconciliation needed: `relationship_type` and `org_id` are categorical, not numeric accumulators.

### 8. Rewire `CITES_DOC` in merge (in `rewire_cross_layer_edges`)

```cypher
MATCH (src:Memory {id: $src_id})-[r:CITES_DOC]->(d:Document)
MATCH (tgt:Memory {id: $tgt_id})
MERGE (tgt)-[:CITES_DOC]->(d)
DELETE r
```

### 9. Hydrate controls and documents after search (in `hydrate_controls_and_documents`)

```cypher
UNWIND $memory_ids AS mid
MATCH (m:Memory {id: mid})
OPTIONAL MATCH (m)-[r:ABOUT_CONTROL]->(c:Control)
OPTIONAL MATCH (m)-[:CITES_DOC]->(d:Document)
RETURN mid,
       collect(DISTINCT {id: c.id, name: c.name, relationship_type: r.relationship_type, org_id: r.org_id}) AS controls,
       collect(DISTINCT {id: d.id, title: d.title}) AS documents
```

Parameters: `$memory_ids` (list of UUIDs from search results).

Called in a separate session.run from the search query, only when `ENABLE_KNOWLEDGE_LAYER=true`. Returns a flat list of records keyed by `mid`. The route assembles the result into a dict for O(1) lookup when building `MemoryHit` objects.

Note on `collect(DISTINCT {...})` with nullable nodes: when no ABOUT_CONTROL or CITES_DOC edges exist, Memgraph's `collect` over a `NULL` node will include a dict with all-null values. The route must filter out dicts where `id IS None` before placing them in `MemoryHit.controls`/`MemoryHit.documents`.

## Existing patterns to reuse

### `merge_memory` ABOUT rewiring (exact source, `memory_repo.py` lines 751–761)

```cypher
MATCH (src:Memory {id: $src_id})-[r:ABOUT]->(x)
MATCH (tgt:Memory {id: $tgt_id})
MERGE (tgt)-[:ABOUT]->(x)
DELETE r
```

`ABOUT_CONTROL` and `CITES_DOC` rewiring uses this exact pattern. No property reconciliation because these edges are not reinforcement-tracked.

### `person_ids` replace pattern (`memory_repo.py` lines 640–655)

```python
session.run(
    "MATCH (m:Memory {id: $id})-[r:ABOUT]->(p:Person) DELETE r",
    id=memory_id,
)
for person_id in patch_fields["person_ids"]:
    session.run(
        """
        MERGE (p:Person {id: $person_id})
        WITH p
        MATCH (m:Memory {id: $memory_id})
        CREATE (m)-[:ABOUT]->(p)
        """,
        person_id=person_id,
        memory_id=memory_id,
    )
```

`replace_control_edges` and `replace_doc_edges` follow this same delete-then-recreate structure.

### Feature flag pattern (`main.py` line 916)

```python
if settings.enable_knowledge_layer:
    from memory_service.knowledge_routes import router as knowledge_router
    app.include_router(knowledge_router)
```

The same `if settings.enable_knowledge_layer:` guard wraps all bridge calls in the routes.

### OPTIONAL MATCH person pattern from `_SEARCH_QUERY_TEMPLATE` (`memory_repo.py` lines 239–241)

```
OPTIONAL MATCH (m)-[:ABOUT]->(per:Person)
WITH m, distance, per
WHERE ($person_ids IS NULL OR per.id IN $person_ids)
```

This shows how OPTIONAL MATCH + WITH + WHERE is chained in the template. The `controls`/`documents` fields are NOT added to the template directly (see rationale in Function Signatures section — hydration is a separate post-search call via the bridge). This avoids adding knowledge-layer Cypher to the core search template when the flag is off, and prevents the template from fanning out on multiple OPTIONAL MATCHes simultaneously.

## Risks / Open Questions

1. **`update_memory` scalar-key exclusion.** The `update_memory` function in `memory_repo.py` (line 622) builds `scalar_keys` by excluding `person_ids` and `strand_ids`. The new fields `control_ids`, `doc_ids`, `control_relationship_type`, and `org_id` must also be excluded from this list — otherwise they will be passed as `SET m.control_ids = ...` which is wrong. The fix is in `main.py` route, not `memory_repo.py`: pass only the scalar fields from `patch_fields` to `memory_repo.update_memory` (strip the bridge fields before the call). This is cleaner than modifying `memory_repo.py`'s exclusion list.

2. **`MemoryClient` pass-through.** `mcp_server/server.py` calls `client.add_memory` and `client.update_memory`. Check `memory_client/client.py` to verify these methods accept and forward `control_ids`, `doc_ids`, `control_relationship_type`, `org_id` as keyword arguments. If not, they need updating too. This file was not read during planning — verify before implementing tasks 9 and 10.

3. **`collect(DISTINCT {...})` with all-null dicts.** When `OPTIONAL MATCH` finds no edges, Memgraph collects `{id: null, name: null, ...}` as a single entry rather than an empty list. The route must filter these before populating `MemoryHit.controls`/`MemoryHit.documents`. This is noted in the Cypher design section but must be tested explicitly (see `test_hydrate_controls_and_documents_filters_null_nodes`).

4. **Dedup reinforce path.** When `add_memory` detects a duplicate and calls `reinforce_memory` instead, `control_ids`/`doc_ids` are silently ignored (the existing memory is reinforced, not a new one created). This matches the behaviour for `strand_ids` today and is intentional — document in code comments.

5. **`org_id` default.** The spec says "defaults to agent's org". The agent node has no `org_id` property in the current schema. Treat `org_id=None` as the default (universal/unscoped) for v1. Do not attempt to look up an org_id from the agent node. Add a TODO comment for future resolution.
