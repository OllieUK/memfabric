# WP-073 Plan: Document Ingestion Pipeline

**Date:** 2026-04-03
**Status:** Ready for implementation

---

## Summary

Create a CLI ingestion script (`scripts/ingest_document.py`) that converts PDF and Markdown files into `Document` + `Chunk` nodes via the HTTP API, with heading-aware (Markdown) and page/character-aware (PDF) chunking, `HAS_NEXT` edges for context expansion, and optional auto-SUPPORTS inference guarded by `INGEST_CHUNK_REVIEW_MODE`. Add two new API endpoints (`POST /knowledge/chunk/supports` and `GET /knowledge/controls/{control_id}/chunks`) with supporting repo functions.

## Pydantic models

### New models in `knowledge_routes.py`

```python
class SupportsCreate(BaseModel):
    chunk_id: str
    control_id: str
    confidence: float          # 0.0–1.0; typically 1 - distance
    status: str = "auto-inferred"  # "auto-inferred" | "confirmed" | "rejected"

class SupportsResponse(BaseModel):
    chunk_id: str
    control_id: str
    confidence: float
    status: str
    created_at: str

class ChunkWithSupports(BaseModel):
    id: str
    text: str
    sequence: int
    doc_id: str
    created_at: str
    confidence: float   # from the SUPPORTS edge
    status: str         # from the SUPPORTS edge
```

`status` is stored as a plain `str` rather than an enum; the valid values are `"auto-inferred"`, `"confirmed"`, and `"rejected"`. This matches the ADR-002 schema without requiring a separate `knowledge_schemas.py` file.

## Function signatures (repo + route)

### `knowledge_repo.py`

```python
def create_supports_edge(
    session,
    chunk_id: str,
    control_id: str,
    confidence: float,
    status: str,
    now: str,
) -> dict:
    """MERGE SUPPORTS edge Chunk→Control; SET all properties on CREATE only.
    Returns {chunk_id, control_id, confidence, status, created_at}.
    Raises if either node does not exist (rely on Memgraph MATCH semantics — no match = silent no-op,
    so the route must validate existence first).
    """

def get_chunks_for_control(
    session,
    control_id: str,
) -> list[dict]:
    """Return all Chunk nodes with a SUPPORTS edge to this control,
    ordered by confidence DESC.
    Returns list of dicts: {id, text, sequence, doc_id, created_at, confidence, status}.
    """
```

### `knowledge_routes.py`

```python
@router.post("/chunk/supports", response_model=SupportsResponse)
async def create_supports(req: SupportsCreate, request: Request) -> SupportsResponse:
    ...

@router.get("/controls/{control_id}/chunks", response_model=List[ChunkWithSupports])
```

Both routes guard existence: `POST /chunk/supports` returns HTTP 404 if either `chunk_id` or `control_id` does not exist (using existing `get_chunk` / `get_control` repo calls before creating the edge).

## Cypher query designs

### `create_supports_edge`

```cypher
MATCH (ch:Chunk {id: $chunk_id}), (c:Control {id: $control_id})
MERGE (ch)-[s:SUPPORTS]->(c)
ON CREATE SET
    s.confidence = $confidence,
    s.status = $status,
    s.created_at = $created_at
RETURN ch.id AS chunk_id, c.id AS control_id,
       s.confidence AS confidence, s.status AS status,
       s.created_at AS created_at
```

Notes:
- `MERGE` on the relationship: re-ingesting the same chunk against the same control is idempotent; the edge is created once and never duplicated.
- `ON CREATE SET` means a pre-existing `confirmed` edge will not be downgraded to `auto-inferred` on re-run.
- `RETURN` is on the `MERGE` clause; this is valid Cypher (no `DETACH DELETE` involved — the Memgraph gotcha does not apply here).
- If either `MATCH` finds no node, the `MERGE` is a no-op and `result.single()` returns `None`. The repo function detects this and raises so the route can return HTTP 404.

### `get_chunks_for_control`

```cypher
MATCH (ch:Chunk)-[s:SUPPORTS]->(c:Control {id: $control_id})
RETURN ch.id AS id, ch.text AS text, ch.sequence AS sequence,
       ch.doc_id AS doc_id, ch.created_at AS created_at,
       s.confidence AS confidence, s.status AS status
ORDER BY s.confidence DESC
```

## Chunking algorithm (pseudocode)

### Markdown path

```
function chunk_markdown(text: str, min_chars: int) -> list[ChunkData]:
    chunks = []
    current_heading = ""
    current_body = []

    for line in text.splitlines():
        if line.startswith("## ") or line.startswith("### "):
            if current_body or current_heading:
                body = "\n".join(current_body).strip()
                candidate = (current_heading + "\n" + body).strip()
                if len(candidate) >= min_chars:
                    chunks.append(ChunkData(text=candidate, heading=current_heading))
            current_heading = line.lstrip("# ").strip()
            current_body = []
        else:
            current_body.append(line)

    # flush final section
    body = "\n".join(current_body).strip()
    candidate = (current_heading + "\n" + body).strip()
    if len(candidate) >= min_chars:
        chunks.append(ChunkData(text=candidate, heading=current_heading))

    return chunks
```

- Section heading is preserved in `text` (prepended to body) so the embedding captures section context.
- `heading` is stored as a metadata field on the Chunk node (see note below on Chunk metadata).
- Documents with no `##`/`###` headings are treated as a single chunk if they meet `min_chars`.

### PDF path

```
function chunk_pdf(path: str, chunk_size: int, overlap: int, min_chars: int) -> list[ChunkData]:
    chunks = []
    full_text = ""

    with pdfplumber.open(path) as pdf:
        for page in pdf.pages:
            page_text = page.extract_text() or ""
            full_text += page_text + "\n"

    # Slide a window of chunk_size with overlap
    start = 0
    while start < len(full_text):
        end = start + chunk_size
        candidate = full_text[start:end].strip()
        if len(candidate) >= min_chars:
            chunks.append(ChunkData(text=candidate, heading=""))
        start += chunk_size - overlap

    return chunks
```

- PDF chunking is character-based with configurable overlap to avoid boundary-cut evidence loss.
- `pdfplumber` is used for text extraction; it handles most standard PDFs without a native build step.

### Note on Chunk metadata (`heading` field)

The existing `Chunk` node in `knowledge_repo.py` stores `text`, `sequence`, `doc_id`, `embedding`, `created_at`. ADR-002 lists `heading`, `body`, `section_ref` as Chunk properties. The `heading` field is not yet in `upsert_chunk`. The ingest script will encode the heading into `text` (prepended) so the embedding captures it. Adding `heading` as a standalone property is deferred — it requires a `ChunkCreate` schema change not scoped to this WP.

If the implementer judges adding `heading` to `ChunkCreate`/`upsert_chunk` to be low-risk, it can be added in this WP, but the plan does not mandate it.

## `scripts/ingest_document.py` structure

The script is self-contained: it reads config from `.env` via `pydantic-settings`, makes HTTP calls to the running API, and prints progress to stdout. It does not import from `memory_service` directly.

### Config class (script-local)

```python
class IngestSettings(BaseSettings):
    api_base_url: str = "http://localhost:8000"
    knowledge_embedding_model: str = "paraphrase-multilingual-MiniLM-L12-v2"
    ingest_chunk_size: int = 2000
    ingest_chunk_overlap: int = 200
    ingest_min_chunk_chars: int = 50
    ingest_auto_supports: bool = False
    ingest_auto_supports_threshold: float = 0.20
    ingest_chunk_review_mode: bool = True

    model_config = SettingsConfigDict(env_file=".env", env_file_encoding="utf-8")
```

### CLI interface

```
python scripts/ingest_document.py <file_path> \
    --doc-id <id> \
    --title <title> \
    --doc-type <policy|procedure|standard|guideline> \
    [--source-url <url>]
```

Uses `argparse`. All required fields are positional or required flags.

### Execution flow

```
1. Parse args, load IngestSettings
2. POST /knowledge/documents  → document_id confirmed
3. Detect file type (suffix: .pdf | .md | .markdown)
4. Chunk the file (markdown_chunker or pdf_chunker)
5. For each chunk (with index as sequence):
   a. POST /knowledge/chunks  (with prev_chunk_id for sequence > 0)
   b. Print progress: "  [OK] chunk {seq}: {len} chars"
6. If INGEST_AUTO_SUPPORTS=true:
   a. For each chunk_id:
      - POST /knowledge/search/controls  {query: chunk_text, limit: 3}
      - If top hit distance < INGEST_AUTO_SUPPORTS_THRESHOLD:
          - candidate = {chunk_id, control_id, confidence=1-distance, status="auto-inferred"}
          - Add to candidates list
   b. If INGEST_CHUNK_REVIEW_MODE=true:
      - Print review summary table (chunk_id | control_id | confidence)
      - Print "Review mode: no SUPPORTS edges created. Set INGEST_CHUNK_REVIEW_MODE=false to apply."
   c. Else:
      - For each candidate: POST /knowledge/chunk/supports
      - Print "Created N SUPPORTS edges (status=auto-inferred)"
7. Print ingestion summary
```

## Risks / Open Questions

| Item | Risk | Mitigation |
|------|------|-----------|
| `pdfplumber` is not in `requirements.txt` | Must be added; ensure it installs cleanly in WSL2 without native build steps | `pdfplumber` is pure Python (wraps `pdfminer.six`); no C extension required |
| MERGE on SUPPORTS edge with `ON CREATE SET` | A `confirmed` edge will not be overwritten on re-ingest (correct behaviour) but could silently succeed without updating properties | Acceptable; the semantics are intentional. Document in script help text. |
| `result.single()` returns `None` when MATCH finds no node | The `create_supports_edge` repo function will detect `None` and the route must raise HTTP 404. The route must call `get_chunk` + `get_control` before the MERGE, or check the MERGE result. | Prefer pre-checking existence with two `get_*` calls (reuses existing helpers); cleaner error messages. |
| Chunk `heading` property absent from current schema | ADR-002 lists it; the existing `ChunkCreate` model does not include it | Plan defers it: heading is prepended into `text`. Flag as a follow-on if needed. |
| `POST /knowledge/chunk/supports` route path collision with chunked IDs | Route is `/chunk/supports` (singular `chunk`) vs `/chunks` (plural) for upsert | Naming is deliberate: `chunk/supports` is a relationship endpoint, not a node CRUD endpoint. Confirm with implementer. |
| Sample PDF fixture for integration tests | Binary file in the test suite | Use `fpdf2` to generate a minimal PDF in the test module's `conftest.py` fixture rather than committing a binary. |
| `argparse` vs `typer` for CLI | Project uses plain `argparse` in existing scripts | Continue with `argparse` for consistency. |
