# WP-074: InfoSec Knowledge Layer — CLI, MCP Tools, and ETL

**Date:** 2026-04-03
**Status:** Ready for implementation

---

## Summary

Expose the knowledge layer to agents (MCP tools), operators (CLI commands), and automated pipelines (bulk ETL script), completing the accessible surface of the knowledge layer. Adds five MCP tools (feature-flag-gated), five CLI subcommands under `memory knowledge`, five `MemoryClient` methods, a validated YAML-driven bulk ETL script, four YAML data files, and a `KNOWLEDGE_LAYER.md` operational runbook.

---

## Approach

The work splits into five independent tracks that can be implemented in parallel once the file-reading step is done:

1. **MemoryClient methods** — add five methods to `memory_client/client.py` that call the existing knowledge HTTP endpoints. No new API surface; these are pure wrappers.
2. **CLI `knowledge` group** — add a `knowledge` Typer sub-app to `memory_client/cli.py`; hang five subcommands off it. Reuse `_make_client()`, error-handling pattern, and `Console`/`Table` rendering from existing commands.
3. **MCP tools** — add five `@mcp.tool` functions at the bottom of `mcp_server/server.py`, wrapped in an `if settings.enable_knowledge_layer:` guard that is evaluated at import time. The `MCPSettings` class in `mcp_server/config.py` does not currently expose `enable_knowledge_layer`; it must be added there (reads from `.env` via `pydantic-settings`, same `extra="ignore"` pattern). Do NOT import from `memory_service.config`.
4. **`scripts/ingest_framework.py`** — standalone ETL script. Reads `.env` via its own `pydantic-settings` `Settings` class (only `API_BASE_URL`). Validates a YAML file against a set of Pydantic models, then calls HTTP API endpoints in dependency order: Framework → Controls → Norms → Documents → Chunks → Jurisdictions → BusinessAttributes. `--dry-run` stops after validation.
5. **YAML data files** — four files under `data/frameworks/`. These are pure data; no code changes required.
6. **`KNOWLEDGE_LAYER.md`** — operational runbook written after the above are complete.

### Dependency order within the ETL script

Upsert in this order to satisfy FK-like constraints:
1. Frameworks (no dependencies)
2. Controls (depend on framework; parent_id optional, but must be created before children if used)
3. Norms (depend on controls via `control_id`)
4. Documents (no dependencies)
5. Chunks (depend on documents via `doc_id`)

Jurisdictions and BusinessAttributes stand alone and can be upserted in any order.

### `review-supports` dependency note

`memory knowledge review-supports` calls `POST /knowledge/chunk/{chunk_id}/supports` (or equivalent) to update edge status to `confirmed` or `rejected`. This endpoint is delivered by **WP-073**, which is listed as a dependency of WP-074. If WP-073 is not yet deployed, the CLI command will receive a 404. The implementation must handle this gracefully: catch `httpx.HTTPStatusError` with status 404 and print a clear message stating that the SUPPORTS edge update endpoint requires WP-073.

### MCP registration guard

`mcp_server/config.py`'s `MCPSettings` class needs one new field:

```python
enable_knowledge_layer: bool = False
```

This reads `ENABLE_KNOWLEDGE_LAYER` from `.env` via `pydantic-settings`. The guard in `server.py` is:

```python
if settings.enable_knowledge_layer:
    @mcp.tool
    def knowledge_search_controls(...): ...
    # ... remaining four tools
```

Because `@mcp.tool` is a decorator applied at import time, the `if` block must wrap the function definitions (not just the decorator calls). This is the established pattern for feature-flagged tools in FastMCP.

## MCP Tool Definitions

All five tools live inside `if settings.enable_knowledge_layer:` in `server.py`. Full docstrings below.

```python
@mcp.tool
def knowledge_search_controls(
    query: str,
    limit: int = 10,
    framework_id: str | None = None,
) -> list[dict]:
    """Search InfoSec controls by semantic similarity.

    Use this tool when an agent needs to find controls relevant to a topic,
    threat, or gap (e.g. "access control for privileged accounts"). Returns
    controls ranked by vector distance to the query, optionally filtered to a
    single framework (e.g. "nist-csf-2.0" or "iso-27001-2022").

    Do NOT use this for searching episodic memories — call memory_search instead.
    Requires ENABLE_KNOWLEDGE_LAYER=true and at least one framework loaded via
    ingest_framework.py.
    """
    with MemoryClient(base_url=settings.api_base_url) as client:
        return client.search_controls(query, limit=limit, framework_id=framework_id)


@mcp.tool
def knowledge_search_chunks(
    query: str,
    limit: int = 10,
    doc_id: str | None = None,
) -> list[dict]:
    """Search policy/procedure document chunks by semantic similarity.

    Use when an agent needs to find specific passages in loaded documents that
    are relevant to a topic (e.g. "data retention requirements" or "incident
    escalation procedure"). Returns chunks ranked by vector distance, optionally
    filtered to a single document by its doc_id.

    Do NOT use this for searching episodic memories — call memory_search instead.
    Requires ENABLE_KNOWLEDGE_LAYER=true and at least one document ingested.
    """
    with MemoryClient(base_url=settings.api_base_url) as client:
        return client.search_chunks(query, limit=limit, doc_id=doc_id)


@mcp.tool
def knowledge_list_norms() -> list[dict]:
    """Return all regulatory norms in the knowledge layer.

    Use when an agent needs the full catalogue of norms to present options to
    the user or to identify which norms apply to a given control. Each norm has
    id, name, text, status, and effective_date.

    This is a catalogue listing, not a search. For semantic search over norm
    text, use knowledge_search_controls (norms are linked to controls via
    IMPLEMENTS edges; searching controls surfaces related norms indirectly).
    """
    with MemoryClient(base_url=settings.api_base_url) as client:
        return client.list_norms()


@mcp.tool
def knowledge_get_control(control_id: str) -> dict:
    """Fetch a single InfoSec control by its ID.

    Use when an agent already has a control_id (e.g. from knowledge_search_controls)
    and needs its full details: name, description, framework_id, and created_at.
    Returns 404 detail if the control does not exist.
    """
    with MemoryClient(base_url=settings.api_base_url) as client:
        return client.get_control(control_id)


@mcp.tool
def knowledge_get_norm(norm_id: str) -> dict:
    """Fetch a single regulatory norm by its ID.

    Use when an agent already has a norm_id (e.g. from knowledge_list_norms)
    and needs its full details: name, text, status, and effective_date.
    Returns 404 detail if the norm does not exist.
    """
    with MemoryClient(base_url=settings.api_base_url) as client:
        return client.get_norm(norm_id)
```

Note: `knowledge_get_control` and `knowledge_get_norm` require two additional `MemoryClient` methods (`get_control` and `get_norm`) that call `GET /knowledge/controls/{id}` and `GET /knowledge/norms/{id}` respectively. These are simple GET wrappers, same pattern as `get_graph`.

## CLI Command Signatures

```
memory knowledge search-controls --query "..." [--limit N] [--framework-id ID]
memory knowledge search-chunks   --query "..." [--limit N] [--doc-id ID]
memory knowledge list-norms
memory knowledge list-documents
memory knowledge review-supports [--framework-id ID]
```

The `knowledge` sub-app is a `typer.Typer` instance added to `app` via `app.add_typer(knowledge_app, name="knowledge")`.

`review-supports` interactive loop:
- Fetch auto-inferred SUPPORTS edges (via a future `GET /knowledge/supports?status=auto-inferred` from WP-073 or WP-075, or directly from the incomplete-jurisdictions endpoint as a proxy)
- For each pair, display chunk text and matched control side-by-side
- Prompt: `[y]es / [n]o / [s]kip / [q]uit`
- `y` → `POST /knowledge/chunk/{chunk_id}/supports` with `{"control_id": ..., "status": "confirmed"}`
- `n` → same endpoint with `status: "rejected"`
- Handle 404 gracefully (WP-073 not deployed)

**Implementation note:** The endpoint for fetching pending SUPPORTS edges and for updating status are both defined in WP-073. The `review-supports` command must be implemented with a clear error path if those endpoints are absent. Consider implementing the command with a stub that prints a helpful message if WP-073 is not yet available, so the rest of WP-074 can be verified independently.

## MemoryClient Method Signatures

```python
def search_controls(
    self,
    query: str,
    *,
    limit: int = 10,
    framework_id: str | None = None,
) -> list[dict]:
    """POST /knowledge/search/controls. Returns list of ControlHit dicts."""

def search_chunks(
    self,
    query: str,
    *,
    limit: int = 10,
    doc_id: str | None = None,
) -> list[dict]:
    """POST /knowledge/search/chunks. Returns list of ChunkHit dicts."""

def list_norms(self) -> list[dict]:
    """GET /knowledge/norms. Returns list of NormResponse dicts."""

def list_documents(self) -> list[dict]:
    """GET /knowledge/documents. Returns list of DocumentResponse dicts."""

def get_incomplete_jurisdictions(self) -> dict:
    """GET /knowledge/incomplete-jurisdictions. Returns diagnostic dict."""

def get_control(self, control_id: str) -> dict:
    """GET /knowledge/controls/{control_id}. Returns ControlResponse dict."""

def get_norm(self, norm_id: str) -> dict:
    """GET /knowledge/norms/{norm_id}. Returns NormResponse dict."""
```

Note: `get_control` and `get_norm` are needed by the MCP tools and are a natural addition alongside the five specified methods.

## YAML Schema (Pydantic models for `ingest_framework.py`)

```python
from pydantic import BaseModel
from typing import Optional

class YamlControl(BaseModel):
    id: str
    name: str
    description: Optional[str] = None
    parent_id: Optional[str] = None   # optional hierarchy within framework

class YamlNorm(BaseModel):
    id: str
    name: str
    text: str
    status: str = "active"
    effective_date: Optional[str] = None
    control_id: Optional[str] = None  # links norm → control via IMPLEMENTS

class YamlDocument(BaseModel):
    id: str
    title: str
    doc_type: str                      # policy | procedure | standard | guideline
    source_url: Optional[str] = None

class YamlChunk(BaseModel):
    id: str
    text: str
    sequence: int
    doc_id: str
    prev_chunk_id: Optional[str] = None

class YamlJurisdiction(BaseModel):
    id: str
    name: str
    region: Optional[str] = None

class YamlBusinessAttribute(BaseModel):
    id: str
    name: str
    description: Optional[str] = None

class YamlFrameworkFile(BaseModel):
    framework_id: str
    framework_name: str
    framework_version: Optional[str] = None
    framework_description: Optional[str] = None
    controls: list[YamlControl] = []
    norms: list[YamlNorm] = []
    documents: list[YamlDocument] = []
    chunks: list[YamlChunk] = []
    jurisdictions: list[YamlJurisdiction] = []
    business_attributes: list[YamlBusinessAttribute] = []
```

Validation runs before any HTTP calls. Any Pydantic `ValidationError` prints the error and exits with code 1.

The ETL script's own `Settings` class:
```python
class ETLSettings(BaseSettings):
    api_base_url: str = "http://localhost:8000"
    model_config = SettingsConfigDict(env_file=".env", env_file_encoding="utf-8", extra="ignore")
```

### Acceptance Criteria

1. `ENABLE_KNOWLEDGE_LAYER=false` (default): no `knowledge_*` tool appears in the MCP tool list. Verified by importing `server.py` and inspecting registered tools.
2. `ENABLE_KNOWLEDGE_LAYER=true`: all five `knowledge_*` MCP tools are registered and callable.
3. `memory knowledge search-controls --query "access control"` returns a Rich table against a live stack loaded with NIST CSF 2.0.
4. `memory knowledge list-norms` prints all norms; `memory knowledge list-documents` prints all documents.
5. `python scripts/ingest_framework.py data/frameworks/nist-csf-2.0.yaml --dry-run` exits 0 with validation output and no DB writes.
6. `python scripts/ingest_framework.py data/frameworks/nist-csf-2.0.yaml` loads NIST CSF 2.0 into a live stack without errors; re-running prints "already existed" for all nodes.
7. `python scripts/ingest_framework.py bad.yaml` exits 1 with a `ValidationError` message before any API calls.
8. `memory knowledge review-supports` without WP-073 deployed prints a clear message indicating the dependency.

## Risks / Open Questions

1. **WP-073 dependency for `review-supports`.** The endpoint to fetch pending SUPPORTS edges and to update their status is not yet in `knowledge_routes.py`. `review-supports` cannot be fully implemented until WP-073 is deployed. Plan: implement the command with a graceful 404 handler and a stub that documents what it will do when WP-073 is live. This satisfies the spec without blocking WP-074 completion.

2. **MCP tool registration with FastMCP.** `@mcp.tool` registers at import time. Conditional registration via an `if settings.enable_knowledge_layer:` block wrapping the function definitions is the correct pattern, but must be tested to confirm FastMCP does not raise on the conditional. If FastMCP requires all tools to be registered at class construction time, an alternative is to use a factory function or register tools programmatically via `mcp.add_tool()`. Verify against the installed FastMCP version before implementing.

3. **`MCPSettings` does not expose `enable_knowledge_layer`.** This is a deliberate separation (MCP server should not import `memory_service.config`). Adding the field to `MCPSettings` is the correct approach; the field reads from the same `.env` key.

4. **YAML data file scope.** The NIST CSF 2.0 and ISO 27001:2022 files may be large (hundreds of controls). The plan calls for representative subsets sufficient for testing — full catalogues can be populated later. The YAML schema must be faithful to the real structure but the data files need not be exhaustive for this WP.

5. **`data/` directory does not yet exist.** Create `data/frameworks/` as part of the implementation. Confirm `data/` is not in `.gitignore` before committing YAML files.
